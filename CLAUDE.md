## Project Overview

**School Store** is a full-stack e-commerce platform built with Next.js 15+ App Router, Supabase (PostgreSQL), and Toss Payments (Korean payment gateway). It supports two user roles — **customer** and **admin** — and is designed for production deployment on Vercel.

## Database Schema

### Tables

| Table | Purpose |
|---|---|
| `profiles` | 1:1 with `auth.users`; stores `role` (`customer` \| `admin`), `full_name`, `phone` |
| `categories` | Product categories (`name`, `slug`, `description`, `image_url`) |
| `products` | Product catalog (`name`, `slug`, `price`, `category_id`, `images`, `is_active`, `is_featured`) |
| `inventory` | 1:1 with `products`; tracks `quantity`, `low_stock_threshold`, `last_restocked_at` |
| `orders` | Customer orders; `status` enum: `pending → paid → processing → shipped → delivered → cancelled → refunded` |
| `order_items` | Line items per order with price snapshot at time of purchase |

### RLS Policies

- All tables have RLS enabled.
- Customers can only read/write their own data.
- Admin access is gated by the `is_admin()` RPC function which checks `profiles.role = 'admin'`.
- Active products (`is_active = true`) are publicly readable without authentication.

### RPC Functions

| Function | Description |
|---|---|
| `is_admin()` | Returns `true` if the calling user has `role = 'admin'` in profiles |
| `decrement_inventory(product_id, quantity)` | Atomically decreases stock; prevents negative inventory |
| `increment_inventory(product_id, quantity)` | Atomically increases stock (for restocks/refunds) |

### Auto-generated Types

`src/types/database.types.ts` is generated from the Supabase schema. **Do not edit this file manually.** Regenerate with:

```bash
npx supabase gen types typescript --project-id <YOUR_PROJECT_ID> > src/types/database.types.ts
```

## Environment Variables

Create a `.env.local` file (not committed — in `.gitignore`):

```bash
# Public — safe to expose to the browser
NEXT_PUBLIC_SUPABASE_URL=https://<project>.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon-key>
NEXT_PUBLIC_TOSS_CLIENT_KEY=test_ck_<...>
NEXT_PUBLIC_SITE_NAME="School Store"
NEXT_PUBLIC_SITE_URL=http://localhost:3000

# Secret — server-side only
SUPABASE_SERVICE_ROLE_KEY=<service-role-key>
TOSS_SECRET_KEY=test_sk_<...>
TOSS_WEBHOOK_SECRET=<webhook-secret>
```

`NEXT_PUBLIC_*` variables are bundled into the client. Never expose `SUPABASE_SERVICE_ROLE_KEY` or `TOSS_SECRET_KEY` to the browser.

## Code Conventions

### Naming

| Artifact | Convention | Example |
|---|---|---|
| React components | PascalCase | `ProductCard`, `OrdersTable` |
| Files (multi-word) | kebab-case | `product-grid.tsx` |
| Hooks | `use` prefix, camelCase | `useCart`, `useAuth` |
| Constants | UPPER_SNAKE_CASE | `TOSS_BASE_URL` |
| Functions / variables | camelCase | `handleSubmit`, `formatPrice` |

### React / Next.js

- **Server Components by default.** Add `"use client"` only when you need browser APIs, event handlers, or Zustand/React hooks.
- Place route-level pages in `src/app/`, shared components in `src/components/`, and feature-specific components in their subdirectory (e.g., `components/admin/`).
- Use the `@/` path alias (maps to `src/`) everywhere — never use relative `../../` imports that cross feature boundaries.

### TypeScript

- **Strict mode is on.** All types must be explicit — avoid `any`.
- Derive DB types from `src/types/database.types.ts`:
  ```ts
  import type { Database } from "@/types/database.types";
  type Product = Database["public"]["Tables"]["products"]["Row"];
  ```
- Use Zod schemas (in `src/lib/validations/`) for all user-facing form validation.

### Styling

- Use Tailwind utility classes. Do not write custom CSS except in `globals.css` for CSS custom properties.
- Use the `cn()` helper from `@/lib/utils` to conditionally merge classes:
  ```ts
  import { cn } from "@/lib/utils";
  className={cn("base-class", isActive && "active-class")}
  ```
- shadcn/ui components live in `src/components/ui/`. Customise them there; never modify the source of a component just to change a one-off style — pass className props instead.

### State Management

- **Cart state:** Zustand store at `src/store/cart.ts` with localStorage persistence. Use `useCart` hook for reads; call store actions directly for writes.
- **Server data:** Fetch directly in Server Components using `createServerClient` from `src/lib/supabase/`. For client-side data that needs caching/refetching, use TanStack Query.

## Supabase Client Usage

Two separate clients exist — use the correct one for each context:

```ts
// In Server Components, Route Handlers, Server Actions:
import { createServerClient } from "@/lib/supabase/server";
const supabase = await createServerClient();

// In Client Components ("use client"):
import { createClient } from "@/lib/supabase/client";
const supabase = createClient();
```

Using the browser client in a Server Component (or vice versa) will cause auth/cookie bugs.

## Payment Flow (Toss Payments)

1. Customer fills checkout form → validated with Zod schema in `src/lib/validations/checkout.ts`.
2. Client calls Toss SDK `requestPayment()` → Toss redirects to `/api/payments/confirm` with `paymentKey`, `orderId`, `amount`.
3. `/api/payments/confirm` calls Toss REST API to confirm the payment, then:
   - Creates an `orders` record.
   - Creates `order_items` records.
   - Calls `decrement_inventory` RPC for each item.
4. Toss sends webhook events to `/api/payments/webhook` — verify the signature using `TOSS_WEBHOOK_SECRET` before processing.

## Admin Panel

Routes under `(admin)/` are protected: server-side checks call `is_admin()` before rendering. Admin capabilities:

- **Products:** Create / edit / deactivate products and manage featured status.
- **Inventory:** View stock levels, update quantities, see low-stock alerts.
- **Orders:** View all orders, update order status.

## Key Patterns & Gotchas

- **Supabase Auth cookies** are managed by `@supabase/ssr`. Do not manually set/read auth cookies.
- **Inventory is decremented atomically** via RPC to prevent race conditions — never update the `inventory` table directly from application code.
- **Price is snapshotted** in `order_items.unit_price` at checkout time — do not reference `products.price` for historical orders.
- **Image URLs** come from Supabase Storage. The domain `*.supabase.co` is whitelisted in `next.config.ts` for `next/image`.
- **Dark mode** uses `next-themes`. Wrap new page layouts in `<ThemeProvider>` if they don't inherit from the root layout.
- **`database.types.ts` is auto-generated.** Any schema migration should be followed by a type regeneration.
